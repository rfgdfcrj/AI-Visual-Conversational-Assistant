### 核心设计原则

| 原则 | 说明 |
|------|------|
| **端侧优先** | 语音识别/合成在浏览器本地完成，零云成本 |
| **按需上云** | 仅在需要视觉理解或复杂推理时才调用云 API |
| **流式响应** | SSE 流式传输，用户感知延迟低 |
| **成本可控** | 运动检测 + 自适应帧率 + 图像压缩，多层成本控制 |

## 3. 运营成本控制策略

这是本项目的核心设计考量之一。AI 视觉对话应用的运营成本主要来自**云 API 调用**（图像 token 消耗大），需要综合运用多种策略将成本控制在可接受范围。

| # | 策略 | 层级 | 预期节省 | 实现状态 | 技术细节 |
|---|------|------|---------|----------|---------|
| C-01 | **运动检测过滤** | 端侧 | 50-80% | ✅ 已实现 | 16×12 缩略图像素差分，仅场景变化时上传 |
| C-02 | **自适应帧率** | 端侧 | 30-50% | ✅ 已实现 | 用户可选 1/2/5 FPS，默认 2 FPS |

| C-04 | **JPEG 有损压缩** | 端侧 | 20-35% | ✅ 已实现 | 质量 60%（可配 30-100%），平衡清晰度与体积 |
| C-05 | **浏览器本地 STT** | 架构 | 100% | ✅ 已实现 | Web Speech API，完全免费，零云成本 |

### 3.2 成本估算模型

以典型使用场景为例（用户连续对话 30 分钟，使用 Qwen-VL-Plus）：

```
假设：
  - 帧率: 2 FPS（每秒 2 帧）
  - 运动检测过滤: 70% 帧被跳过（用户大部分时间坐在桌前）
  - 实际发送帧: 2 × 60 × 30 × 30% = 1,080 帧
  - 每帧 token: ~800 tokens（512px JPEG 压缩后）
  - 用户文本: ~50 tokens/轮 × 30 轮 = 1,500 tokens
  - AI 回复: ~200 tokens/轮 × 30 轮 = 6,000 tokens
  - 模型: Qwen-VL-Plus (~$0.3/$0.9 per MTok)

  图像 input: 1,080 × 800 = 864,000 tokens → $0.26
  文本 input: 1,500 tokens → $0.0005
  AI output: 6,000 tokens → $0.005
  ─────────────────────────────
  30 分钟会话总费用 ≈ $0.27

  （对比：不带运动检测时 3,600 帧 → $0.87，节省 69%）
  （对比：使用国外厂商同等 VL 模型 → $13+，Qwen 便宜 98%）
```

## 4. 技术选型与决策

### 4.1 为什么选择 Web 应用而非原生 App？

| 维度 | Web 应用 | 原生 App |
|------|---------|----------|
| 摄像头访问 | ✅ MediaDevices API | ✅ AVFoundation/CameraX |
| 语音识别 | ✅ Web Speech API（免费） | ✅ 原生 SDK（需额外集成） |
| 部署成本 | ✅ 零安装，一个 URL | ❌ 需应用商店审核 |
| 跨平台 | ✅ 一次开发，全平台 | ❌ iOS + Android 分别开发 |
| AI API 集成 | ✅ 直接 HTTP 调用 | ✅ 但需处理平台限制 |
| 开发速度 | ✅ 快速迭代 | ❌ 编译-部署周期长 |

**决策**：Web 应用作为 MVP 是最优选择。后续可包装为 PWA 获得离线能力和系统级体验。

### 4.2 AI 模型选择

| 模型 | 适用场景 | 成本 (per 1M tokens) |
|------|---------|---------------------|
| **Qwen-VL-Plus** | 日常对话、物体识别（推荐） | ~$0.3 input / ~$0.9 output |
| **Qwen-VL-Max** | 复杂场景分析、专业理解 | ~$3.0 input / ~$9.0 output |
| **Qwen2.5-VL-72B** | 开源旗舰 VL 模型 | ~$1.5 input / ~$4.5 output |

**策略**：默认使用 Qwen-VL-Plus（推荐经济模式），用户可根据需要手动切换到更高级模型。

**重要优势**：Qwen VL 系列原生支持视觉理解，一个模型同时处理"看图"和"对话"，不需要额外模型或复杂的图像预处理管道。

### 4.3 浏览器兼容性

| 功能 | Chrome | Edge | Firefox | Safari |
|------|--------|------|---------|--------|
| getUserMedia | ✅ | ✅ | ✅ | ✅ |
| SpeechRecognition | ✅ | ✅ | ❌ | ⚠️ (部分支持) |
| SpeechSynthesis | ✅ | ✅ | ✅ | ✅ |
| ES Modules | ✅ | ✅ | ✅ | ✅ |

**推荐浏览器**：Chrome 或 Edge（Chromium 内核，SpeechRecognition 支持最好）。

## 6. 已知限制与未来路线图

### 6.1 已知限制

| 限制 | 影响 | 缓解措施 |
|------|------|---------|
| Web Speech API 在 Firefox/Safari 兼容性差 | 部分用户无法使用语音 | 提供文字输入备选方案 |
| 运动检测对光线变化敏感 | 光线变化可能产生误触发 | 可调灵敏度参数 |
| 视频帧消耗大量 token | 成本高于纯文本对话 | 多层压缩 + 运动检测 |
| 无离线能力 | 断网时完全不可用 | 未来考虑 WebLLM |
