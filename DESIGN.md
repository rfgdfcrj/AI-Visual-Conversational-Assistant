### 1.2 用户故事详述

#### P0 — 核心功能（5/5）

**US-1: 摄像头开启**
- 用户点击"开启摄像头" → 浏览器请求权限 → 视频流显示在预览区
- 验收：Chrome/Edge/Safari 均可用，权限拒绝时有中文提示

**US-2: 语音识别**
- 用户点击"开启麦克风" → 说话 → 实时转录显示 → 说完自动发送给 AI
- 验收：识别结果准确，中间结果实时显示，自动重启 continuous 模式

**US-3: 语音播报**
- AI 回复后自动朗读（默认关闭，可手动开关）
- 验收：中文 TTS 正常，播报时暂停麦克风防止回声

**US-4: 文字聊天**
- 用户输入文字 → Enter 发送 → AI 流式回复
- 验收：SSE 流式显示，支持 2000 字符输入限制

**US-5: 视觉识别**
- 用户展示物品 → 点击快照分析 → AI 描述画面内容
- 验收：Qwen VL 模型正确识别物体、场景、文字、人物

#### P1 — 重要功能（8/8）

**US-6: 模型切换**
- 用户从下拉菜单选择 Plus/Max/72B 模型
- 验收：模型名白名单校验，切换后立即生效

**US-7: 运动检测**
- 摄像头以 2 FPS 捕获，比较前后帧差异，静止时跳过上传
- 验收：画面静止时运动指示器显示"静止"，无 API 调用

**US-8: 多对话管理**
- 侧栏显示对话列表，新建/切换/删除，localStorage 持久化
- 验收：刷新页面对话不丢失，上限 50 个对话、500 条消息/对话

**US-9: 自然对话风格**
- AI 像朋友一样聊天，口语化、幽默、场景化
- 验收：不出现"作为AI助手"等机械表达，回复 2-4 句为主

**US-11: 手机适配**
- 900px/640px/500px/375px 四断点，安全区域，触控友好
- 验收：iPhone SE 到 iPad 均无布局破裂，按钮可手指点击

**US-12: 对话持久化**
- 所有对话存 localStorage，刷新恢复
- 验收：配额溢出时自动降级保留最近 10 个对话

**US-13: 数据安全**
- API 端点不可配置，Origin/Referer 校验，速率限制
- 验收：跨域请求返回 403，超频请求返回 429

**US-14: 防并发**
- AI 处理中禁用发送/快照/语音输入
- 验收：连续点击只发起一次请求，不叠加 Token 消耗

#### P2 — 锦上添花（2/3）

**US-10: 动态表情**
- AI 回复完成后，根据关键词匹配对应 emoji，弹跳动画切换
- 验收：12 条关键词规则覆盖主要场景

**US-16: 公开分享**
- 部署到 Railway，生成公开 URL
- 验收：任何人可通过链接访问

**US-17: 文字识别** → ⚠️ 依赖 Qwen VL 模型原生能力，前端不额外处理

#### P3 — 未来规划（0/3）

**US-18: 跨设备同步** — 需后端数据库 + 用户认证，架构变更较大
**US-19: 长期记忆** — 需向量数据库或摘要机制，Token 成本需评估
**US-20: 对话导出** — 前端可实现，计划 v2.1

## 二、运营成本控制

### 2.1 成本分析

DashScope Qwen VL 按 token 计费：

| 模型 | 输入 (每百万 token) | 输出 (每百万 token) |
|------|---------------------|---------------------|
| qwen-vl-plus | ~$0.30 | ~$0.90 |
| qwen-vl-max | ~$3.00 | ~$9.00 |
| qwen2.5-vl-72b | ~$1.50 | ~$4.50 |

**图像 token 消耗**：Qwen VL 将图像按分辨率转换为 token（约 1 token / 28×28 像素块）。一张 512px 的图像约消耗 300-400 tokens。

### 2.2 策略对照：想到的 vs 采用的

| # | 策略 | 类型 | 状态 | 预估节省 | 说明 |
|---|------|------|------|---------|------|

| 2 | 图像缩放到 512px | 减少单次 Token | ✅ 已采用 | 60-80% | 前端 Canvas 缩放 + JPEG 60% 质量 |
| 3 | 语音识别浏览器本地 | 零成本 | ✅ 已采用 | 100% | Web Speech API，不消耗云服务费 |
| 4 | 语音合成浏览器本地 | 零成本 | ✅ 已采用 | 100% | SpeechSynthesis API，不消耗云服务费 |
| 5 | 对话历史限制 | 减少输入 Token | ✅ 已采用 | 40-60% | MAX_HISTORY=20，超出自动截断 |
| 6 | 模型分级默认 Plus | 选择便宜模型 | ✅ 已采用 | 90% | 默认 qwen-vl-plus，用户手动升级 |

| 15 | 图像数量限制 | 控制单次成本 | ✅ 已采用 | 不定 | 每次最多 5 张，每张最大 ~2MB |
| 16 | 压缩质量可调 | 用户权衡 | ✅ 已采用 | 10-30% | 设置面板可调 JPEG 质量 30%-100% |
| 17 | CDN 缓存静态资源 | 减少带宽 | ❌ 未采用 | — | Railway 自带 CDN，不需额外配置 |
| 18 | 离线 PWA | 减少重复加载 | ❌ 未采用 | — | Service Worker 未配置，计划 v2.1 |

**统计**：想到 18 项策略，已采用 12 项，未采用 6 项。

### 2.3 单次对话成本估算

**典型场景**：用户开启摄像头，说 5 句话，AI 回复 5 次

| 项目 | 用量 | 单价 | 费用 |
|------|------|------|------|
| 文本输入 token | ~500 tokens × 5 轮 = 2,500 | $0.30/1M | ~$0.0008 |
| 图像输入 token | ~350 tokens × 5 张 = 1,750 | $0.30/1M | ~$0.0005 |
| 输出 token | ~200 tokens × 5 轮 = 1,000 | $0.90/1M | ~$0.0009 |
| **合计** | | | **~$0.0022** |

即：**一次完整对话约 0.2 美分**。即使每天 100 次对话，月成本约 $6.60。

**无运动检测的最坏情况**（摄像头一直开着，画面不断变化）：

| 项目 | 用量 | 费用 |
|------|------|------|
| 每秒 2 帧 × 60 秒 × 10 分钟 = 1,200 张图像 | 1,200 × 350 tokens | ~$0.13 |
| 加上文本对话 | | ~$0.002 |
| **10 分钟连续使用** | | **~$0.13** |

运动检测可将此降低 70-90%，实际约 $0.01-0.04。

### 2.4 成本监控

- 前端实时显示累计 Token 用量和预估费用
- 服务端日志记录每次 API 调用的状态码
- 速率限制防止单用户异常消耗
- Railway 部署可设置用量告警

## 三、技术决策记录

### 3.1 为什么选 SSE 而非 WebSocket？

- SSE 单向流（服务端→客户端）满足 AI 流式回复需求
- 实现更简单，Express 原生支持
- 自动重连，不需要心跳维护
- WebSocket 适合双向实时通信，本场景不必要

### 3.2 为什么用 localStorage 而非 IndexedDB？

- 数据量小（对话文本，不含图像）
- API 简单，JSON 直接读写
- 配额溢出有降级策略
- IndexedDB 适合更大数据量（如离线图像缓存），当前不需要

### 3.3 为什么语音识别/合成用 Web Speech API？

- 完全免费，浏览器本地处理
- 中文识别质量可接受
- 不需要额外 API Key 或服务
- 缺点是 Safari 支持有限（仅 TTS），但 Chrome/Edge 完整支持

### 3.4 为什么后端做代理而非前端直调 DashScope？

- 保护 API Key 不暴露到前端
- 可以做输入校验、速率限制、Origin 校验
- 可以清洗错误信息防止泄露后端细节
- 未来可以加缓存层减少重复调用
